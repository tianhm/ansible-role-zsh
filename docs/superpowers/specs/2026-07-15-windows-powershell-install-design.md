# Windows (PowerShell + cmd) installer — design

Date: 2026-07-15
Status: Approved (design), pending implementation plan

## Problem

The role installs `zsh + antigen + starship + fzf` on Linux/macOS via Ansible,
driven by `install.sh`. zsh cannot run in native Windows `cmd.exe` or
PowerShell. Users want to bring the *portable* parts of this setup — chiefly the
starship prompt and the fzf/autosuggestion/history shell UX — to native Windows
shells, without WSL, without Ansible, and without Python.

## Goal

Deliver a **"feels like our zsh" experience on native Windows PowerShell**, plus
the starship prompt in `cmd.exe`, via a standalone PowerShell installer that
relies on a preinstalled package manager (winget / scoop / choco).

Non-goals:

- Running zsh or this role's zsh config on Windows (that path is WSL/Git Bash and
  is out of scope).
- Driving the Windows install through Ansible.
- Porting the ~25 zsh antigen completion bundles.

## Decisions (from brainstorming)

| Decision | Choice |
|---|---|
| Scope | Prompt + shell UX (not prompt-only, not full-zsh-via-MSYS2) |
| Architecture | Standalone `install.ps1`, no Ansible / no Python / no WSL |
| Target shells | PowerShell (5.1 and 7+) **and** cmd.exe via clink |
| Package managers | Auto-detect `winget → scoop → choco`; require one; fail if none |
| Plugins | Curated Windows feature set with per-feature on/off flags; unsupported bundles not ported |
| Config delivery | Ship a pre-rendered static `starship.toml` (no Jinja on Windows) |

## Feature mapping (zsh → Windows)

| zsh feature | Windows equivalent | cmd.exe? |
|---|---|---|
| starship prompt | starship (native, cross-platform) | yes (via clink) |
| zsh-autosuggestions | PSReadLine `-PredictionSource History` | no |
| fast-syntax-highlighting | PSReadLine built-in token coloring | no |
| fzf Ctrl+R history / fzf-widgets | fzf + PSFzf module (Ctrl+R, Ctrl+T) | no |
| git-extras / docker / kubectl / … completions | posh-git + native argument completers | no |
| ~25 antigen completion bundles | **not ported** (PowerShell has its own completion model) | — |

cmd.exe receives **only** the starship prompt; PSReadLine and PSFzf are
PowerShell-only.

## File layout

```
install.ps1              # entry point (curl|iex friendly, also runs from a checkout)
windows/
  starship.toml          # pre-rendered from templates/starship.toml.j2 defaults (static)
  starship-init.lua      # clink script for cmd.exe
  Windows.Tests.ps1      # Pester tests (profile-merge idempotency)
```

`install.ps1` lives at the repo root, parallel to `install.sh`, so the
`irm | iex` one-liner is discoverable.

## Install flow (`install.ps1`)

1. **Detect package manager** — probe `winget → scoop → choco`. If none found:
   print install links for all three and `exit 1`.
2. **Install binaries** with the found manager: `starship`, `fzf`, and `clink`
   (unless `-NoCmd`).
3. **Install PS modules** from PSGallery, `-Scope CurrentUser`: `PSFzf`,
   `posh-git`, and ensure `PSReadLine >= 2.2` where possible. Each failure →
   warn, skip that feature, continue.
4. **Deliver config** — copy `windows/starship.toml` (from checkout) or download
   it from raw GitHub (piped run) to `%USERPROFILE%\.config\starship.toml`.
   Skip if it already exists unless `-Force`.
5. **Patch the PowerShell profile** — write a managed marker block into
   `$PROFILE.CurrentUserAllHosts` (idempotent, see below).
6. **cmd support** (unless `-NoCmd`) — write `starship-init.lua` into
   `%LocalAppData%\clink\`.
7. **Summary** — print per-feature status (✓ installed / ⚠ skipped) and
   "restart your shell."

### Remote-install one-liners

```powershell
irm https://raw.githubusercontent.com/viasite-ansible/ansible-role-zsh/master/install.ps1 | iex
# with flags:
& ([scriptblock]::Create((irm https://raw.githubusercontent.com/viasite-ansible/ansible-role-zsh/master/install.ps1))) -NoCmd -NoPoshGit
```

### Flags

- `-NoCmd` — skip clink install and cmd.exe configuration.
- `-NoAutosuggestions` — skip PSReadLine prediction lines.
- `-NoPSFzf` — skip PSFzf module + fzf key bindings.
- `-NoPoshGit` — skip posh-git.
- `-Force` — overwrite an existing `starship.toml`.
- `-PackageManager <winget|scoop|choco>` — force a specific manager (default: auto-detect).

## Managed profile block

Wrapped in markers so re-running replaces only the managed region and leaves the
user's own profile content intact:

```powershell
# >>> viasite-ansible zsh >>>
# managed block - do not edit between the markers; regenerated by install.ps1
$env:STARSHIP_CONFIG = "$HOME\.config\starship.toml"
Invoke-Expression (&starship init powershell)              # prompt

Set-PSReadLineOption -PredictionSource History             # autosuggestions   (-NoAutosuggestions)
Set-PSReadLineOption -PredictionViewStyle ListView         #   (PSReadLine >= 2.2 only)

Import-Module PSFzf                                         # fzf Ctrl+R / Ctrl+T (-NoPSFzf)
Set-PsFzfOption -PSReadlineChordProvider 'Ctrl+t' -PSReadlineChordReverseHistory 'Ctrl+r'

Import-Module posh-git                                      # git completion    (-NoPoshGit)
# <<< viasite-ansible zsh <<<
```

Each line is emitted only when its flag is on **and** the underlying
module/binary is actually present (so a skipped module never produces a broken
profile that errors on every shell start).

## clink script (cmd.exe)

`windows/starship-init.lua`, dropped into `%LocalAppData%\clink\`:

```lua
load(io.popen('starship init cmd'):read("*a"))()
```

## Error handling

- **No package manager** → hard fail with winget/scoop/choco links, `exit 1`.
- **Module install fails** → warn, mark the feature skipped, continue; reflected
  in the final summary.
- **Old PSReadLine** (Windows PowerShell 5.1 ships 2.0, no predictions) → try to
  install 2.2+; if the in-use copy cannot update, degrade (omit the prediction
  lines) rather than error.
- **ExecutionPolicy** → document the `-ExecutionPolicy Bypass` one-liner; the
  script does not change machine-wide policy itself.
- **Idempotent** → safe to re-run; `-Force` re-copies `starship.toml`.

## Testing & config-drift guard

- **Pester test** (`windows/Windows.Tests.ps1`) for the profile-merge function —
  pure string logic, runs under `pwsh` on any OS (usable in existing CI, not only
  Windows).
- **GitHub Actions `windows-latest` job** — runs `install.ps1` end-to-end,
  asserts `starship --version` works, `%USERPROFILE%\.config\starship.toml`
  exists, and the managed block is present in the profile.
- **Drift check** — a CI step renders `templates/starship.toml.j2` with default
  vars and diffs the result against `windows/starship.toml`, failing if they
  diverge (keeps the static Windows config honest against the Jinja source of
  truth).
- The Linux `molecule/` suite is untouched.
